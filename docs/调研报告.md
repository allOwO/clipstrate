# ChopClip 调研报告

> 日期：2026-07-21
> 背景：现用剪贴板工具 iCopy 存在三个痛点 —— ① UI 老旧、不符合 macOS 26 Liquid Glass 设计语言；② 内存占用过高；③ 缺少文本分词（拆词）能力，无法快速选取淘口令、邮箱、手机号等片段复制。本报告调研竞品现状、关键技术可行性与交互参考，为新工具 ChopClip 的规划提供依据。

---

## 1. 竞品分析

| 产品 | 价格 | UI 观感 | 内存表现 | 分词/实体提取 | 备注 |
|---|---|---|---|---|---|
| **iCopy**（现用） | 免费+内购 | 老旧，仍是 macOS 10.x 时代风格 | 偏高（用户主诉痛点） | ❌ 无 | 主打剪贴板历史 + 快捷回复短语，MAS 4.8 星，兼容 macOS 10.14+（包袱重，无法用新 API） |
| **Maccy** | 免费开源 | 极简（菜单栏列表 / 搜索弹窗） | 最优，公认轻量 | ❌ 无 | Swift + SwiftData(SQLite)，纯本地无遥测；43k 条历史下 SQLite 依然快，瓶颈在 UI 渲染而非存储（重要经验，见 §3） |
| **Paste** | 订阅制 | 精美，卡片流 | 偏高，有崩溃/占用抱怨 | ❌ 无 | iCloud 同步、Pinboard，视觉最强但重 |
| **PasteNow** | 买断 | 现代原生 | 中等 | ❌ 无 | 国人开发，中文友好 |
| **Raycast 剪贴板** | 免费（附属功能） | 现代 | 宿主本身较重 | ❌ 无 | 依附于 Raycast 启动器 |
| **PopClip** | 买断 | 选中文本浮条 | 轻 | ❌（有扩展但非分词） | 交互参考价值：选中文本即弹浮动操作条 |

**结论：macOS 上没有任何主流剪贴板工具提供"大爆炸式"中文分词选取能力，这是明确的差异化空白。** 「Liquid Glass 原生观感 + Maccy 级轻量 + 分词拆选」三点组合即是 ChopClip 的定位。

## 2. macOS 26 Liquid Glass 设计语言

- SwiftUI 提供 `.glassEffect()` 修饰符（AppKit 对应 `NSGlassEffectView`），原生实现硬件加速的模糊、半透明、折射高光效果，支持 tint、形状、交互态参数。
- 多个玻璃元素应放入 `GlassEffectContainer`：系统自动融合相邻玻璃形状、统一光照，并支持形变过渡（morphing）动画，同时提升渲染性能——适合"面板弹出/词块炸开"这类过渡。
- 系统自动处理「减少透明度」等辅助功能回退；深浅色模式自适应。
- 要点：玻璃是"浮在内容之上的功能层"，不应滥用于内容本身；剪贴板面板、拆词面板正是典型的玻璃层场景。
- **直接以 macOS 26 为最低部署目标**可全量使用这些 API，无需自绘玻璃或兼容旧系统（iCopy 兼容 10.14 正是它 UI 无法翻新的原因）。

## 3. 内存占用：根因与对策

竞品内存高的常见原因及对应策略：

1. **全量加载历史到内存/视图**。Maccy 的 issue 数据表明：数万条记录下 SQLite 本身不慢，慢和占用都发生在"打开 UI 时一次性取出并渲染所有行"。→ 对策：列表懒加载 + 分页查询，UI 只物化可见行。
2. **图片原图驻留内存**。截图类条目动辄数 MB。→ 对策：原图落盘（文件目录 + 数据库存路径），内存只持有小缩略图，预览时按需解码。
3. **富文本/大文本整段保留**。→ 对策：列表只存前 N 字符摘要，全文按需读盘。
4. Electron/混合框架天然高占用 → ChopClip 用纯 Swift/SwiftUI 原生实现即可规避。

**量化目标（写入规划）：常驻（菜单栏空闲）< 40 MB，面板打开峰值 < 100 MB；对照组 Maccy 常驻约 20–40 MB。**

## 4. 剪贴板监听与隐私新政（关键风险）

- macOS 没有剪贴板变更通知 API，业界标准做法是 **定时轮询 `NSPasteboard.general.changeCount`**（0.2–0.5s 间隔），计数变化才真正读内容。changeCount 比较本身开销极低。
- ⚠️ **macOS 15.4 起预览、26 全面启用的剪贴板隐私机制**：应用在无用户粘贴交互时程序化读取剪贴板会触发系统弹窗；`NSPasteboard.accessBehavior` 允许用户对每个 App 设置「始终允许 / 询问 / 拒绝」。
  - 对剪贴板管理器的影响：首启必须引导用户将本 App 设为「始终允许」，否则轮询会反复弹窗，产品不可用。这是所有剪贴板管理器都要过的坎，需要做成 onboarding 的一等公民。
  - 新增的 `detect` 系 API 可在**不读取内容**（不触发弹窗）的情况下探测剪贴板上数据的类型模式，可用于优化探测逻辑。
- 行业惯例：尊重 `org.nspasteboard.ConcealedType`（密码管理器标记的机密内容）与 transient 类型，**不入库**。

## 5. 中文分词与实体识别技术选型

### 5.1 分词引擎

| 方案 | 优点 | 缺点 |
|---|---|---|
| **NLTokenizer**（系统 NaturalLanguage 框架，`unit: .word`） | 原生、离线、零依赖、零额外内存包袱；对简中/繁中/中英混排开箱即用 | 无自定义词典；新词、专名切分偏保守 |
| jieba 系移植（swift-jieba 等） | 可自定义词典、粒度可调 | 需内置词典文件（数十 MB 内存/包体），维护成本高 |
| 在线分词 API（锤子开源版 Big Bang 的做法） | 质量高 | 联网、隐私不可接受，直接排除 |

**结论：首选 NLTokenizer**（符合"轻量"目标），架构上把分词抽象为 `Segmenter` 协议，保留日后替换/叠加词典的余地。分词质量的不足用「粒度切换（字/词/自动合并）+ 实体层兜底」在交互上弥补——用户真正高频复制的是实体（号码/口令/邮箱），实体靠规则识别，不依赖分词准确率。

### 5.2 实体识别（拆词面板的"智能层"）

两级识别，在分词结果之上做高亮与聚合：

- **系统 `NSDataDetector`**：URL、电话、地址、日期、航班——免费且质量好。
- **自定义正则**（对中文场景补强）：
  - **淘口令**：货币/特殊符号包裹的 8–14 位字母数字，如 `￥ABCdef12345￥`；实际变体多（`$…$`、`€…€`、`((…//`、数十种符号），采用「符号集 + 中段 token」的宽松匹配再校验长度。
  - 中国手机号 `1[3-9]\d{9}`、座机、邮箱、6/4 位验证码（结合上下文词"验证码/code"提权）、快递单号（SF/JD/YT 等前缀规则）、身份证 18 位、银行卡（Luhn 校验）、IP/MAC、UUID/Git SHA、金额。
- 实体在面板中呈现为**预选高亮的智能块**，排在普通词块之前，支持数字键一键复制（详见规划中的交互设计）。

## 6. 拆词交互参考

- **锤子 Smartisan OS「大爆炸（Big Bang）」**（交互原型的黄金标准）：长按文本 → 词块"炸开"平铺 → 点选/划选多个词块 → 底部操作（复制/搜索/分享/翻译）。已开源（Android），其核心启发：① 词块保持原文顺序流式排布，空间关系即语义关系；② **划动刷选**比逐个点击快得多；③ 选中即时高亮聚合。
- **微信「长按消息 → 拆词」**：验证了移动端拆词选取在中文用户中的习惯养成——目标用户对这个交互零学习成本。
- **PopClip**：macOS 上「选中文本 → 原位浮条」的成熟范式，证明非激活浮动面板（不抢焦点）在 Mac 上体验成立。
- **获取全局选中文本的技术路径**（用于"对任意 App 选中文本直接拆词"入口）：优先 Accessibility API（`AXUIElementCreateSystemWide` → `kAXFocusedUIElement` → `kAXSelectedText`），对不支持 AX 的 App（浏览器部分区域、Electron）回退为模拟 ⌘C 读剪贴板再恢复现场。PopClip、各划词翻译工具均为此方案，成熟可行。需要「辅助功能」权限。

## 7. 总体结论

1. **差异化成立**：macOS 缺少"分词拆选"型剪贴板工具；三个痛点（玻璃 UI、轻量、拆词）技术上全部可行且互不冲突。
2. **最低目标定 macOS 26**：换取原生 `glassEffect` 全家桶与最简实现。
3. **两个必须前置处理的平台风险**：剪贴板隐私弹窗（onboarding 引导"始终允许"）＋ 辅助功能权限（划词入口需要）。
4. **轻量的三板斧**：纯 SwiftUI 原生、图片落盘只留缩略图、列表懒加载分页。
5. **分词不追求完美**：NLTokenizer 打底，交互上用粒度切换与实体识别层兜底，把"复制淘口令/手机号/邮箱"做成一键动作。
