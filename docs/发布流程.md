# Clipstrate 发布流程

## 目标

推送符合 `v主版本.次版本.修订号` 的 Git tag 后，GitHub Actions 在原生
arm64 `macos-26` runner 上自动构建，并创建同名 GitHub Release。

发布产物：

- `Clipstrate-<版本>-macOS-arm64.zip`
- `Clipstrate-<版本>-macOS-arm64.dmg`
- `Clipstrate-<版本>-macOS-arm64.sha256`

版本号取自 tag，例如 `v0.1.0` 会写入 App 的
`CFBundleShortVersionString = 0.1.0`；`CFBundleVersion` 使用 GitHub Actions
的递增运行编号。

## 版本号约束（硬约束，违反 = 分不清测试与正式版）

- **正式发布**用干净语义化版本号（如 `0.2.0`），tag 为 `vX.Y.Z`。
- **测试 / 预发布 / 临时验证**构建必须带可区分尾缀，**绝不与正式版同号**：
  - 预发布 tag：`vX.Y.Z-alpha.N` / `-beta.N` / `-rc.N`（流水线触发条件 `v*` 天然兼容）；
  - 本机临时构建：在 `MARKETING_VERSION` 或产物文件名上加尾缀（如 `Clipstrate-0.3.0-beta.app`）。
- 交付测试包后主动清理，不在 `/Applications`、`~/Applications`、桌面等处留**同名副本**——同名同版本会让 Launch Services 启动到错误的一份，也让人无法区分。

## 发布

先确保待发布提交已经位于远端 `main`，然后创建并推送 annotated tag：

```bash
git tag -a v0.1.0 -m "Clipstrate v0.1.0"
git push origin v0.1.0
```

推送后，`.github/workflows/release.yml` 会：

1. 校验 tag 格式；
2. 通过 XcodeGen 生成工程；
3. 用 Release 配置构建 macOS arm64 App；
4. 做 ad-hoc 代码签名并验证签名；
5. 生成 ZIP、DMG 和 SHA-256；
6. 上传 Actions artifact；
7. 创建 GitHub Release 并自动生成变更说明。

tag 是不可变的发布标识。发布内容需要修正时应增加版本号，不移动或覆盖旧 tag。

## 不发布的打包验证

在 GitHub 仓库的 Actions → Release → Run workflow 中填写版本号（例如
`0.1.0`），可以完整执行构建和打包并下载 Actions artifact，但不会创建 tag
或 GitHub Release。

本机也可以复用同一个脚本：

```bash
./scripts/package-release.sh v0.1.0 dist 1
```

## 当前签名限制

当前不使用付费 Apple Developer Program，因此流水线只能进行 ad-hoc 签名，
不能做 Developer ID 签名和 Apple 公证。下载后的 App 可能被 Gatekeeper 提示
“无法验证开发者”，用户需要在 Finder 中右键 App 并选择“打开”进行确认。

Hardened Runtime 已在 Release 配置开启（公证前置条件；Debug 保持关闭，否则库
校验会拒绝 XCTest 注入导致测试宿主崩溃）。将来取得 Developer ID 证书后，仅需在
现有 Release workflow 中补齐：

1. 从 GitHub Actions Secrets 导入 Developer ID Application 证书，改用其签名；
2. 使用 `notarytool` 上传并等待公证；
3. 对 App/DMG 执行 stapling，再发布产物。
